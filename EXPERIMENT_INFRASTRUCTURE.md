# Experiment infrastructure: what each component protects

This document is a collaborator-facing map of the experimental setup. It
explains why each component exists, what conclusion it can support, and what
must connect to it before the result becomes research evidence. The detailed
research design is in [PLAN.md](PLAN.md); implemented and remaining work is
tracked in [IMPLEMENTATION.md](IMPLEMENTATION.md); commands and artifact links
are collected in [README.md](README.md). Exact current artifact identities are
kept in the [artifact identity ledger](ARTIFACT_IDENTITY_LEDGER.md) so this
guide can use unambiguous human-readable names.

The most important current fact is:

> **There is no model-quality result yet.** The checked-in model runs, data
> products, benchmark families, runtime probes, and canaries are engineering
> evidence only. Relevant candidate-execution, research-training,
> model-selection, scored-evaluation, and claim-authorization flags remain
> false.

This is deliberate. The project evaluates programs generated by a model, so a
credible score depends on much more than a working trainer. The benchmark must
measure the intended semantics, the candidate must run inside a reviewed
boundary, the data must not leak across lifecycle stages, the deployed model
size must be accounted correctly, and the statistical contrast must be fixed
before sealed results are opened.

## 1. The claims determine the infrastructure

The project has two separate research lanes.

1. **Fixed-size specialization** asks whether post-training can improve Unix
   terminal performance while architecture, physical parameter count,
   serialized precision, and deployed weight size remain unchanged.
2. **Compression** asks whether task-aware pruning, width or layer reduction,
   vocabulary trimming, factorization, distillation, or quantization can
   improve the terminal-performance frontier versus physical parameters or
   deployed bytes.

These lanes must remain separate because a method that improves accuracy with
the same model answers a different question from one that preserves accuracy
in a smaller artifact. Quantization can establish byte, memory, or latency
compression, but not a reduction in physical parameter count. The evaluation
therefore records parameters, serialized bytes, effective precision, peak
memory, and runtime performance separately instead of collapsing them into one
ratio.

The default object of study is a **dense, non-MoE model below one billion total
physical parameters**. A mixture-of-experts appendix is triggered only if the
dense study first finds reproducible, meaningfully distinct capability
clusters. Even then, shared weights, every expert, the router, embeddings, and
the output head must together remain below one billion parameters, and expert
ablation and routing tests must demonstrate separate expertise. This prevents
the label “expert” from concealing a larger or weakly differentiated network.

Ability loss is not a goal. It matters only when an above-floor ability can be
shown, through controlled removal and add-back, to trade against better target
performance or a smaller deployment footprint. Safety-oriented suppression or
forgetting with unchanged performance and size is outside the success
criteria.

## 2. The evidence flow

The infrastructure is a chain of commitments and re-opened evidence, not a
collection of interchangeable JSON files:

```text
PLAN + immutable campaign policy
                 |
                 v
       prospective run specifications <------ pinned model/data inputs
                 |                                      |
                 v                                      v
       paired campaign registry              admitted corpus + token schedule
                 |                                      |
                 +---------------------> training + update/FLOP ledger
                                                        |
                                                        v
                                             exported model inspection
                                                        |
sealed benchmark contract ---> isolated evaluator <----+
          |                         |
          v                         v
  trusted fixture/oracle      bound per-task outcomes
          |                         |
          +-------------------------+
                                    v
                       paired statistics + claim gates
```

Engineering canaries sit beside this path. They test whether a mechanism is
feasible, but they do not enter the claim path merely because they pass. Each
claim-bearing step must reopen and validate the exact source artifacts named by
the next step; copying a digest into a record is not independent verification.

## 3. Plans, provenance, and campaign profiles

[PLAN.md](PLAN.md) fixes the scientific question, operator funnel, lifecycle
splits, required baselines, and success thresholds. The machine-readable
[campaign policy](configs/campaign-policy.json) translates the repeated parts
of that plan into immutable screening, confirmation, and runner-up profiles.
Prospective runs validate against
[run-spec.schema.json](run-spec.schema.json), and cross-run structure validates
against [campaign-registry.schema.json](campaign-registry.schema.json).

This layer matters because most accidental research flexibility is introduced
before training: changing a seed after seeing a failure, giving a favored arm
more tuning, reversing reference and comparison roles, choosing a different
checkpoint rule, or silently spending extra tokens. A prospective run spec
commits to the model and tokenizer, data identities, target/support mixture,
operator and dose, seeds, training protocol, token and FLOP ceilings, and
intended export before results exist. The campaign registry then checks the
complete roster rather than validating runs in isolation: replicate counts,
paired data order, fresh confirmation seeds, direct-baseline relationships,
teacher-corpus identity, and evaluation coverage must agree across the whole
campaign.

Hashes provide tamper evidence, not timestamp evidence. A party able to
rewrite every prospective file and every dependent record could construct a
different internally consistent campaign. Claim-bearing run-spec hashes must
therefore also be published, timestamped, or entrusted to an external
preregistration mechanism before execution.

## 4. Executable Bash benchmarks and their lifecycle

The primary endpoint is not textual similarity to a reference command. It is
whether one generated Bash program produces the correct output and filesystem
state on every hidden fixture for a semantic specification. A specification is
defined by its operator/dependency graph, utility composition, filesystem
schema, and output contract. Splitting at those semantic levels is important:
different wording around the same program is leakage, not generalization.

The static suite has distinct training, operator-selection, method-development,
shadow-validation, sealed in-distribution, and sealed compositional-OOD roles.
Training may shape weights; operator-selection and method-development may
shape the method; shadow validation selects a checkpoint; sealed suites are
opened once after method and analysis lock. A separate bounded-interactive
suite tests whether static synthesis gains transfer to a small terminal loop,
but the static functional pass rate remains primary.

Benchmark artifacts have four importantly different states:

| State | What it is for | What it does **not** establish |
|---|---|---|
| Public semantic scaffold | Exercises split generation, normalized graphs, hashes, and lifecycle routing | Concrete executable fixtures, independent verifiers, genuine OOD, or a score |
| Family-local development data | Develops one executable operator family and its edge cases before additive integration | Membership in a frozen suite or independent production approval |
| Frozen public development catalog | Gives collaborators an immutable, reproducible method-development identity | Confidentiality, sealed status, model-selection authority, or claim eligibility |
| Sealed evaluation assets | Provide hidden prompts and fixtures after generator, verifier, parser, sandbox, and analysis lock | Nothing yet: the claim-eligible sealed assets have not been completed or opened |

## 5. Trusted oracles, property verification, and verifier audits

An executable benchmark is only as strong as the code that decides “correct.”
For each concrete family, a trusted oracle derives the expected semantic
result and a property/state verifier checks the candidate workspace. Whenever
possible, a separately structured reference implementation must reach the same
answer without sharing the production oracle's reasoning path.

This separation matters because shared bugs can give every candidate the same
false score. Mutation tests intentionally corrupt output bytes, modes, links,
paths, records, ordering, or state and require rejection. Descriptor-relative,
no-follow access and before/after stability checks reject path swaps observed
during inspection. A trusted supervisor must first stop and reap candidate
descendants and establish workspace quiescence; sequential scans alone cannot
exclude a concurrent mutation after the final observation. A stratified human
audit is still required before sealing because implementation agreement cannot
prove that the encoded task matches the intended natural-language contract.

The verifier also checks the complete output policy, not only a preferred
file. Unexpected files, altered inputs, forbidden metadata changes, dangling
links, or surviving background processes can be functional failures even if
one output line looks right. This is why a reference-program string match or a
unit test with one example is insufficient.

The fifth `pipefail-atomic-report` family illustrates the boundary between a
semantic requirement and an observable outcome. Its trusted constructions
model complete logical streams and an ordered vector of configured stage
statuses, then require the exact success report, failure report, rollback
bytes, or absent output selected by the policy. Checked-in tests exercise both
constructions, all 100 fixtures, randomized valid streams, and final-state
mutations without executing a candidate program. The workspace verifier can
establish exact final files, modes, links, inputs, and report bytes only after
trusted quiescence. It cannot infer whether publication actually used an
atomic rename, statuses came from Bash `PIPESTATUS`, the claimed pipeline
topology was executed, or only allowed tools were invoked. Its fixtures
likewise do not cover explicit directory permission errors or live effective-
access failures, and sequential scans do not prove global quiescence.

The sixth `bounded-retry-state-machine` family makes policy control flow
observable without claiming to observe a candidate's internal execution. Its
event ledger distinguishes success, transient failure, ordinary failure, and
terminal failure. The five policies impose one-, two-, four-, or six-attempt
per-state-visit limits; fixed and until-terminal policies retry transient and
ordinary failures, transient-only stops on ordinary failure, and terminal
failure always stops retrying. Budgets reset for each state/visit. The exact
attempt and terminal reports expose branch choice, bounded revisits,
compensation, empty input, missing events, and causes. Separately structured
parsers, simulators, cause builders, and serializers must agree. The
final-state verifier can check those bytes and the exact tree only after
trusted quiescence. It cannot attest
actual retry, waiting, state traversal, compensation, allowed-tool use, atomic
publication, transient input preservation, global quiescence, or candidate
exit status; explicit directory-permission and live effective-access failures
also remain outside the fixtures.

The seventh `case-routed-batch-transform` family combines a manifest loop,
exclusive case routing, byte-exact transforms, and batch-level fallback
behavior. Four route keys cross five fallback policies. Two separately
structured parsers, routers, transform implementations, and serializers must
agree on status/error records and the route-partitioned output tree. The
workspace verifier authenticates inputs and the complete final state only
after trusted quiescence. It cannot attest actual route choice, transform
execution, read scope, allowed-tool use, atomic publication, transient input
preservation, global quiescence, or candidate exit status; explicit directory-
permission and live effective-access failures also remain outside the fixtures.

The eighth `collision-safe-batch-rename` family recursively discovers regular
candidates without following symlinks, derives a flat destination under one of
four rename rules, and applies one of five collision policies. Two separately
structured parsers, destination functions, grouping/planning engines, and
ledger serializers must agree on the immutable per-source action plan and
exact outputs. Unlike input-preserving families, its mutation-aware verifier
expects moved and coalesced sources to be absent while retained leaves must
equal their authenticated baseline. Original input directories must preserve
kind, mode, and link topology; their size and modification time are correctly
nonsemantic after child removal. Every output file must match the selected
representative's bytes, size, mode, and authenticated-baseline modification
time. These checks require trusted quiescence. They cannot establish actual
rename or inode identity, collision-decision or read-scope history, allowed-
tool use, staging or atomic publication, crash rollback, transient input
preservation, global quiescence, or candidate exit status; explicit directory-
permission and live effective-access failures also remain outside the fixtures.

The ninth `hardlink-deduplicated-mirror` family requires real inode sharing,
not merely equal bytes. Four equivalence keys cross five deterministic owner
policies, with explicit probes that make all 20 cells observably different.
Input fixtures include pre-existing aliases and committed mtimes. Two
separately structured parsing and grouping paths must agree before shared
final-state assembly, and the verifier checks exact bytes, modes, mtimes,
input preservation, link counts, portable hardlink-group digests, and the
complete ledger. It requires trusted quiescence and cannot establish creation,
tool, transient-path, global-quiescence, or exit-status history. Its fixed
reviewed Bash canary proves only public-development feasibility for the
declared tool set.

The tenth `compressed-archive-roundtrip-verify` family separates a canonical
oracle representative from semantic candidate acceptance. The trusted path
decodes one bounded stream under the selected gzip, bzip2, xz, or uncompressed
contract and parses the resulting ustar without extracting untrusted names.
It rejects unsafe or noncanonical member paths, wrong order or metadata,
links and extensions, duplicate or unexpected members, malformed padding, and
data beyond the terminator. The report is relational: its digests are derived
from the actual candidate archive and parsed members, not copied from the
oracle's representative compressed bytes. The workspace verifier additionally
requires exact reconstructed bytes/modes/zero mtimes, link count one, output
closure, and stable inputs. These checks require trusted quiescence and cannot
establish actual verification steps, tool use, operation order, temporary
paths, causal reconstruction history, global quiescence, or candidate exit
status. The reviewed all-fixture Bash canary is feasibility evidence only.

## 6. Parser, decoding, and deterministic outcomes

The response parser is frozen before evaluation and accepts only raw Bash or
one optional Markdown fence. Decoding limits, token limits, syntax policy,
timeouts, fixture order, allowed tools, rerun rules, and exclusions are part of
the prospective [evaluation specification](evaluation-spec.schema.json).

This component matters for two reasons. First, parser changes can manufacture
performance improvements without changing the model. Second, a single “fail”
bucket hides whether the model failed or the infrastructure failed. Bound
[task-result records](task-result.schema.json) preserve extraction, syntax,
tool-policy, fixture, timeout, output-overflow, and terminal outcomes, together
with per-invocation resource maxima and an execution-only retry chain. Scored
selection must return exactly one terminal outcome per semantic task. Task
exclusions currently fail closed because a post-hoc exclusion mechanism would
otherwise be an easy route to optimistic scores.

Deterministic decoding does not imply deterministic CUDA training. The former
makes evaluation repeatable; the latter is separately reported and never
silently assumed.

## 7. Sandbox, runtime closure, and the supervisor

Generated shell is untrusted code. A correct-looking benchmark score is not
credible if a candidate can read host files, access the network, escape its
workspace, interfere with later fixtures, fork past a timeout, or exploit a
different utility version. Every scored invocation therefore needs a fresh
rootless namespace with a read-only root, isolated writable workspace, no
network or host mounts, no added capabilities, no privilege escalation, fixed
identity and locale, and enforced PID, CPU, memory, file-descriptor, output,
and wall-time limits.

Pinning the container image is necessary but not sufficient. “Runtime closure”
means identifying every executable, loader, shared library, dynamically opened
resource, locale/configuration input, and allowed tool that can affect the
program. Otherwise a supposedly reproducible test may depend on mutable host
state. The current runtime-manifest, materialization, sealed-memfd snapshot,
descriptor-handoff, and systemd/Bubblewrap namespace work progressively tests
this chain without reopening mutable source paths.

A trusted supervisor/PID1 is equally important. It must install child seccomp,
measure cumulative CPU rather than only a CPU rate, bound and classify output,
reap all descendants, establish workspace quiescence, and enforce the exact
tool policy. A host-side timeout or `CPUQuota` alone does not prove these
properties. Until the Bash runtime-data/`dlopen` closure, externally trusted
launcher, independent namespace validation, and supervisor gates are complete,
the sandbox cannot execute synthesized candidates for research scoring.

The native lifecycle canary now tests the supervisor mechanics separately from
candidate semantics. A reproducibly built, sealed static program runs as PID1
and receives only nine fixed binary requests covering normal exit,
double-fork/`setsid`, zombie cleanup, wall timeout, both stream overflows, CPU
fan-out, a forbidden syscall, and child-emitted frame-spoof bytes. It applies a
fixed child-only seccomp filter, kills and reaps the namespace process set,
sums `wait4` resource usage, confirms that PID1 alone remains, and returns an
authenticated request-bound record. The outer controller also kills/stops the
unique transient unit, requires it to report inactive/dead with no cgroup path,
and synchronously reaps the wrapper. Running it inside the same rootless
namespace and user-systemd envelope is important because lifecycle bugs are
not meaningfully tested by an in-process mock.

That canary intentionally has no candidate, command, workspace, fixture,
verifier, or score input. It therefore validates a design seam without turning
development machinery into a security claim. The fixed filter cannot stand in
for a Bash/runtime policy; measured cumulative CPU is not yet an enforced
ceiling; local binary hashes are not external trust; and the canary does not
hold a scored workspace quiescent. Those distinctions keep every general
execution and claim-authority flag false while making the next implementation
step smaller and testable.

The next seam connects exactly one reviewed case without opening a candidate
API. A private development case binds an exhaustively admitted first-catalog
fixture to a source-reviewed Bash response. Its candidate-input-free controller
reconstructs that case, a host-pinned Bash/find/sort/mkdir runtime, and one
fixed policy; rebuilds the checked-in native supervisor; and launches the
reviewed program through the same systemd/Bubblewrap shape. The exact binary
protocol binds the invocation, program, fixture definition, initial workspace,
runtime, tool policy, resource limits, process outcome, captured-stream
digests, separate `wait4` totals, cumulative CPU including live namespace-tree
observations, descendant reaping, and a bounded post-run workspace projection.

Native PID1 omits the top-level `input` subtree from that projection because
the fixture deliberately contains unreadable inputs. After it has killed and
reaped descendants and the controller has locally verified an inactive, empty
cgroup through the pinned systemctl observation, it seals and parses the
snapshot, independently revalidates the descriptor-pinned input baseline,
compares the output-side tree, and invokes the existing property verifier. The
canary therefore tests one complete fixed-case integration seam. Its public
execution entry point accepts only an optional nonce and no caller-selected
program, command, fixture, runtime, or verifier; it permanently denies
arbitrary-candidate, scoring, model-selection, and claim authority. The public
snapshot projection omits
raw payload bytes but includes paths,
modes, sizes, and payload digests. It is not answer-confidential and must not
cross a sealed-evaluation boundary or become benchmark feedback.

Runtime-data/`dlopen` closure, external
trust, a general synthesized-Bash seccomp policy, and exact-tool enforcement
also remain unresolved.

## 8. Dense model inspection and sub-1B accounting

The local Safetensors inspector reads a flat artifact without importing model
code. It checks shard layout and stable file identity, accounts tensor storage,
classifies components, records tokenizer sources, rejects suspicious custom
loading hooks, and distinguishes evidence for dense versus MoE structure. A
bounded local-only forward probe separately establishes basic runtime
compatibility.

This is essential because repository labels and configuration fields do not
prove model size. Tied/shared storage must be counted once physically but
reported clearly at the logical architecture level; embeddings and output
weights cannot be omitted; quantization scales, zero points, codebooks, and
padding contribute deployed bytes; and unstructured zeros do not make a
smaller deployable model unless a supported runtime realizes the saving.
Architecture-specific shape reconstruction and completed-record binding are
still required before the generic inspection can prove a valid dense sub-1B
campaign export.

If the MoE appendix ever triggers, accounting covers total physical storage,
not active parameters per token. Meaningfully separate expertise must then be
shown by held-out routing and expert-ablation effects, not inferred from router
names or sparse activation.

## 9. Training corpus admission and token schedules

The data path intentionally separates these steps:

1. pin and replay the raw source;
2. deduplicate and preserve provenance;
3. audit licensing, contamination, ambiguity, utilities, and candidate safety;
4. admit individual rows for the declared target/support role;
5. tokenize with the exact student tokenizer;
6. build the deterministic target/support schedule and packed sequences;
7. train while recording actual visible and supervised tokens per update.

A content-addressed corpus is reproducible, but not automatically licensed,
correct, safe, decontaminated, or useful. The current conservative NL2SH-ALFA
lexical survivors are therefore static candidates, not admitted research
training data. The checked-in
[data pilot](reports/engineering-data-pilot/manifest.json) demonstrates source
replay and exact scheduling only.

Schedules distinguish target tokens, support-replay tokens, non-padding input
tokens, and response-plus-EOS supervised tokens. This prevents nominal
“20M-token” arms from receiving different optimization exposure through
padding, prompt length, packing, or a partial accumulation. The campaign's
80% target / 20% replay design isolates capability support from simply adding
more target data. All-retain and minimal-support arms spend the same total
replay budget; one-family add-backs test whether a lost capability actually
mediates a target or footprint tradeoff.

The optimizer ledger records learning rate, actual token counts, gradient
norm, checkpoint identity, and measured FLOPs for every update. Teacher data,
when used, is generated only from training assets, verified by execution, and
shared identically between comparable arms. Equal-target-token and
equal-total-FLOP views are both required so extra selection, calibration, or
teacher work cannot masquerade as a better method.

## 10. Operator funnel, baselines, and selection

The project does not assume that SwiGLU channels are the right unit. The
fixed-size funnel compares dense SFT, sequence distillation, minimal-support
replay, selected reset/regrow, no-reset tuning, and random controls. Candidate
units include residual branches, attention-head groups, FFN blocks, consistent
hidden dimensions, and embedding/output groups. Short cross-fitted look-ahead
updates test prospective terminal learnability before one granularity can be
promoted. SwiGLU side-tensor recycling becomes the main mechanism only if FFN
blocks win that comparison.

The compression funnel separately compares layer removal, structured
attention/FFN pruning, hidden-width reduction, vocabulary trimming,
task-aware mixed precision, factorization, and native smaller-model
distillation. It keeps nondominated accuracy-versus-byte configurations and
allows only one preregistered two-operator hybrid. This limits search-induced
overfitting.

Baselines identify the explanation for a gain. Dense extra-step controls test
whether more compute suffices; random removal tests generic regularization;
no-reset tuning tests whether reinitialization matters; minimal-support dense
SFT tests whether ordinary replay explains specialization; uniform
quantization tests target-aware bit allocation; task-agnostic pruning tests
target-specific structure; and a native smaller dense model tests whether a
complicated compression method beats simply starting smaller.

The capability-support audit occurs before any “irrelevant ability” choice.
Audited language, programming, factual, reasoning, and prose families must be
above behavioral floor, and protected terminal prerequisites remain protected.
A family is called dispensable only after selective decline, target
non-inferiority, and a same-budget add-back demonstrate the predicted tradeoff.

## 11. Completed records, registries, and evaluation binding

The prospective run spec says what should happen. A completed
[experiment record](experiment-manifest.schema.json) says what actually
happened and binds the realized operator, token/FLOP ledgers, checkpoint rule,
model export, and inspection identity back to the prospective digest. The
campaign registry then joins every required replicate and evaluation rather
than accepting a convenient subset.

For supported floating-dense Qwen2/Qwen3/Llama outputs, a separate
completed-model evidence companion now freshly reinspects the source and
export, reconstructs both exact dense inventories, and reconciles the
completion's model identity and accounting. It semantically validates saved
runtime reports and binds their aggregate storage, class, and vocabulary
projections to the fresh static identities, but it does not rerun or
authenticate those observations. Supported pruning also has to change the
planned layer, uniform FFN-width, or uniform all-layer Qwen3
complete-GQA-group head-width architecture dimension. The companion still
cannot prove which selected source indices or
values populated the export, that training consumed the pinned source bytes,
or that factorized/quantized/hybrid formats are correct. It is permanently
nonauthorizing and must be reopened by a later claim binder rather than merely
copied into one.

The evaluation spec binds one inspected export to one benchmark inventory,
ordered fixture set, decode/parser policy, sandbox limits, and prospective
analysis commitment. Per-task results bind every observed outcome back to that
inventory. The outcome builder reopens the registry and result collections and
derives the paired binary arm-by-seed-by-task cube; it does not accept a
caller-supplied summary table.

This chain prevents several otherwise invisible substitutions: scoring the
wrong checkpoint, mixing task versions, dropping a failed seed, changing the
comparison direction, reusing a screening seed in confirmation, or attaching
hardware measurements from a different export. The example records under
[`examples/`](examples/) validate document shape only and are not real
provenance.

## 12. Statistics and claim acceptance

The primary statistic is macro functional pass@1 over semantic
specifications, not over individual fixtures. Arms are paired by training
seed, data order, teacher corpus, task, and fixture. The analysis uses a crossed
seed/task bootstrap, paired randomization tests, Holm adjustment across the two
confirmatory lanes, and simultaneous non-inferiority checks for bounded
terminal and protected capabilities.

The thresholds are fixed in the plan before sealed evaluation. In brief, a
fixed-size winner needs a meaningful static gain over the strongest matched
dense baseline with the same artifact footprint and replicated terminal
non-inferiority. A compression winner needs a material byte reduction at
near-equal performance, or a meaningful performance gain at matched bytes,
and must beat the strongest task-agnostic compression baseline while reducing
measured memory. Both lanes require fresh seeds, runner-up-backbone
replication, and claim-eligible independent benchmarks.

The implemented claim-policy layer is intentionally fail closed. It can list
which statistical, export, compute, hardware, replication, and teacher-free
conditions would be required, but `claim_authorized` remains false until every
projection is independently rederived from its named source artifacts. A
passing p-value alone cannot authorize a claim about size, compute matching,
capability preservation, or deployment.

## 13. Engineering canaries are not research evidence

Canaries answer narrow questions cheaply before the expensive campaign:

- the GPU pilot asks whether each candidate backbone can be inspected, loaded,
  and updated on the local RTX 5090;
- the corpus and schedule pilot asks whether raw bytes can be replayed into an
  exact token ledger;
- the dense-SFT canary asks whether real-text optimization, update hashing,
  export, and reopening work end to end;
- descriptor and namespace canaries ask whether sealed runtime bytes can cross
  specific process and mount boundaries under bounded control.
- the case-routed Bash canary asks whether one fixed, source-reviewed program
  can realize all 100 public fixture contracts with a restricted tool `PATH`.
- the collision-safe rename Bash canary asks whether one fixed, source-reviewed
  program can realize all 20 rule/policy cells on the binary profile under a
  restricted tool `PATH`, including exact arbitrary-byte equality.
- the nested-JSON migration canary asks whether one fixed Bash wrapper and
  isolated standard-library Python program can realize all 100 strict
  shape/policy/profile contracts under the declared three-tool `PATH`.

Their reports are in
[`reports/engineering-pilot/`](reports/engineering-pilot/),
[`reports/engineering-data-pilot/`](reports/engineering-data-pilot/), and
[`reports/engineering-dense-sft-canary/`](reports/engineering-dense-sft-canary/).
These artifacts contain no terminal-performance, capability, sealed-benchmark,
or model-selection result. Locally hashed executables are not automatically
externally trusted, and a fixed probe reporting that isolation worked is not
independent attestation of its own environment. The namespace canary therefore
does not authorize synthesized-candidate launch even when its transport and
parser behave as designed. Likewise, the case-routed canary is not a caller-
selected candidate API, production sandbox, observation of tool or routing
history, scored evaluation, model-selection result, or research claim.
The rename canary has the same boundary: its all-byte/NUL probes and successful
final states do not establish rename, inode, collision-decision, read-scope,
tool, staging, atomic-publication, crash, or quiescence history and confer no
execution, scoring, selection, or claim authority.

Keeping this boundary explicit lets us use canaries aggressively for
engineering without accidentally upgrading feasibility evidence into a
scientific conclusion.
